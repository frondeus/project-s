Here the problem is, that by using 
`(self :a)` we can only infer that `self` is "applicable" to `:a`. 

It is because we have three ways of "applicability":
* Function application - aka function call.
* Object application - aka field access.
* Array application - aka index access.

All three are done by the same syntax:

so when have `(a b)`, we can only tell that
`a` is applicable to `b` without knowing what `a` or `b` are.

Even when we have `(a :b)`, we can only tell that `a` is applicable to `:b` because who knows maybe its just a function `a` that takes keyword `:b` as an argument.

It works for the type checking, because that's all we need.
But when we want to describe what is the type of the function, suddenly we have no way of expressing that.

It would have to be something like

```rs
fn<Self, Root, Super, Origin>(Self, Root, Super, Origin) -> Self
where
  Self: Applicable(:a)
```

if we had similar syntax for types and boundaries.
Which we obviously do not have and is quite complex.
Then, maybe just maybe we could say that actually `obj/con` is adding extra constraint to that fn

So we could infer that:

```rs
fn<Self, Root, Super, Origin>(Self, Root, Super, Origin) -> Self
where
  Self: Applicable(:a) & {} // where {} means "is object"
```

And then maybe just maybe we could actually detect such a combination
and merge it into:

```rs
fn<Self, Root, Super, Origin>(Self, Root, Super, Origin) -> Self
where
  Self: { :a }
```

```type
(refmut<{}> & Applicable(:a -> Any), Any, Any, Any) -> Any
```

Then we have another issue to solve.
`obj/insert` is a function, yes, but the one that modifies the object.
Which means, `Self` always stays empty from our point of view.
At least when we entering the function.

So technically what we are doing here is:


```ignore

      self

        (self)

            :a)))))
```

If i had `obj/insert` to return `self`.

Typechecker here must be aware of thunks and that they are lazy.

# Structs


Okay. Cool. So I think it makes sense to actually separate structs and objects.

Structs would be plain values, without the access to `self` or `root` or `super` or `origin`.

of course I already took `obj/struct` for creating objects, and for now i dont want to change it.
Lets create separate function `obj/plain`


```
(obj/plain
  :a 1
  :b 2
)
```

```processed
(do
  (obj/plain
    :a
    1
    :b
    2))
```

```type
{a: 1, b: 2}
```

Perfect!

Okay. So how can we add field to plain object?
This "kinda" works but its of course overriding object.

```

(set t (obj/plain :a 1))

```

This type is bit weird, but makes sense. We are not "merging" different use heads into one.

```type
ref<{} | {a: 1}, mut {} & {a: 1}>
```

```json

  "a": 1.0
}
```

Therefore, lets for now forget about refs in terms of "adding" a field.
Instead, we ad plain struct extensions:

```
(let :t (obj/plain :a 1))

(obj/extend t :b 2)
```

```json

  "a": 1.0,
  "b": 2.0
}
```

```type

```

However it might be a challenge to step from plain structs to objects with thunks.
