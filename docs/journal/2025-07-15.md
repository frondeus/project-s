In order to fix splices i need to think what I want to achieve.

First idea i have - you can only use one splice per application?

Like, splice can be at the end of the application.

Second idea - maybe that constraint can be relaxed.
So what is needed?

* Only lists and tuples can be spliced - that makes sense.
* Tuples are easier - they have fixed size.

So if our destination tuple (btw, its always tuple)

has (x , y, z, w) shape,
we can have (1, ..(2, 3), 4). That should work just fine.

Also if there is a shape of
(x, y, z, w, ..r)

then we can have rest pattern in any size as long as there are minimum 4 elements.

From type inference perspective how could we implement rest pattern?

The most sensible way would be to have some kind of tuple concatenation:

`(1, ..(2, 3), 4)`
could be represented as `(1,) + (2, 3) + (4,)`.

Then we are getting certain finished tuple `(1, 2, 3, 4)` that we can compare to `(x, y, z, w)`.

In case of lists, since we support `..[x]` and we dont know the size of that list, we can only allow rest pattern to be at the end of the concatenation
so its `(1,) + (2, 3) + (4,) + [x]` and that would be represented as `(1, 2, 3, 4, ..x)`

Does it make sense?
